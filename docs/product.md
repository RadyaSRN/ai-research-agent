# Продуктовая гипотеза

Документ раскрывает пункт 1 требований: формулирует гипотезу с AS IS → TO BE и прикидывает эффект.

## Целевой пользователь

Исследователь (индустриальный или академический), в основном, ориентированный на ML, для которого ежедневный трекинг новых публикаций — часть рабочего процесса. В типичную неделю такой исследователь:

- ведёт несколько одновременных исследовательских проектов (2–5);
- внутри каждого проекта держит в голове несколько "открытых" идей/направлений, под которые хочется следить за литературой;
- отслеживает минимум одну-две arXiv-категории (`cs.LG`, `cs.AI`, `cs.CL`, `cs.CV` и смежные);
- периодически ищет код к статьям — на GitHub, в Papers with Code, в репах авторов.

## AS IS

Исследователь вручную открывает arXiv (listing или RSS), просматривает 30–100 заголовков и abstract'ов в день, фильтрует глазами по текущим проектам, складывает интересное в закладки / Notion / `.md`-файлы / почту самому себе. На это уходит **30–60 минут в день**.

Наблюдаемые проблемы такого процесса:

- **Временны́е затраты.** Полчаса-час каждый день — это 10–20 часов в месяц на одну активность, которая не создаёт value сама по себе, а только фильтрует вход.
- **Отсутствие персональной памяти.** Статьи, которые исследователь уже видел (но решил не читать полностью), забываются за 2–3 недели. Когда неделю спустя обнаруживается, что в проекте нужен именно этот результат, статью приходится искать заново по туманным ключевым словам.
- **Периферийные темы выпадают.** Когда проектов несколько, глаза привыкают сканировать список по доминирующей теме — и смежные идеи (которые иногда и дают breakthrough) проскакивают мимо.
- **Альтернативы отстают по свежести.** Semantic Scholar, Connected Papers и аналоги индексируют arXiv с задержкой от нескольких дней до недель; для быстро движущихся областей (RL, LLM reasoning, diffusion) это критично.
- **Нет связи между поиском статей и поиском кода.** Исследователь отдельно смотрит arXiv, отдельно лезет в GitHub искать реализации — два разных workflow'а в двух разных вкладках.

## TO BE

Исследователь ведёт свои проекты и идеи в Telegram-боте диалогом; бот хранит их в персональной БД вместе с эмбеддингами. Каждое утро (по настраиваемому расписанию и таймзоне) в Telegram приходит дайджест из 3–5 свежих arXiv-препринтов по каждой активной идее, с оценкой релевантности и коротким LLM-обоснованием, почему статья подобрана.

Ключевые свойства TO BE:

- **Автоматическая ингестия arXiv.** Cron раз в сутки тянет новые препринты из целевых категорий, апсертит в локальный корпус (`papers`), параллельный worker досчитывает эмбеддинги. Свежесть корпуса — не хуже, чем у arXiv-listing'а (т.е. сутки, а не дни-недели Semantic Scholar).
- **Накапливающийся личный корпус.** Все когда-либо встреченные статьи остаются в БД и доступны для семантического поиска. Вопрос "а как называлась та работа про token-level entropy в RL, я её видел недели три назад?" решается одной фразой в чат.
- **Ретривал под идеи, а не под запрос.** Идея, которую исследователь один раз сформулировал ("energy-based CFG в активациях"), автоматически используется как анкер для поиска всё новых релевантных работ — каждый день заново, без повторного формулирования запроса.
- **Двухуровневый retrieval: cosine + LLM-реранкер.** Грубый отбор делается дёшево по эмбеддингам, финальный — LLM-судьёй, который читает текст идеи и abstract статьи и выставляет оценку релевантности + reasoning. См. оффлайн-оценку качества: [`evals.md`](evals.md).
- **Поиск по GitHub из того же интерфейса.** Главный агент ходит во внешний GitHub MCP, поэтому переход от статьи к её коду не требует переключения контекста.
- **Эмбеддинги идей и проектов — задел на будущее.** Сейчас они используются только для retrieval в дайджесте; по этой же инфраструктуре позже можно делать дедупликацию идей, предлагать связи между проектами, автоматически подсказывать "смотри, эта статья ложится сразу в две твои идеи".

Примерные временны́е затраты TO BE:

- 3 минуты на чтение утреннего дайджеста (просмотр 5–10 карточек с обоснованиями);
- 2–4 минуты в день на редкое управление идеями/проектами через чат.

**Итого ≈ 5–7 минут в день против 30–60 минут AS IS.**

## Гипотеза

> Автоматизация ежедневного литературного обзора через персональный дайджест на базе retrieval + LLM-ранжирования снижает временны́е затраты исследователя на трекинг arXiv в 6–10 раз и одновременно не ухудшает охват релевантной литературы за счёт (1) персональной памяти по уже виденным работам и (2) явного учёта нескольких параллельных идей.

Гипотеза понятная, измеримая (таймы до/после, recall по контрольному набору статей) и фальсифицируемая — если охват ухудшится или экономия составит меньше трёх раз, гипотеза не подтвердится.

## Оценка эффекта

Простая прикидка по одному исследователю:

| Метрика                    | AS IS       | TO BE          | Δ              |
|---|---|---|---|
| Минут в день               | 45          | 7              | −38 мин         |
| Часов в месяц (22 раб. дн.)| 16.5        | 2.6            | **−13.9 часов** |

При ставке junior-researcher ~1 500 ₽/час — **~21 000 ₽/мес** на одного пользователя.
При ставке senior ~3 000 ₽/час — **~42 000 ₽/мес**.

**Методология оценки.** Simple math, цифры времени — экспертная оценка (прошлый собственный опыт автора + стандартная продолжительность, которую обычно называют коллеги). Замеров на реальных пользователях "до / после" в рамках этого прототипа не проводилось; это естественный следующий шаг при масштабировании (набрать 5–10 исследователей, замерить таймы и recall на контрольной выборке препринтов).

**Потенциальный эффект.**

- **Высокий** для исследователей, у которых ежедневный трекинг литературы — формально часть рабочей нагрузки (PhD-студенты, research engineers в R&D командах). Для них −14 часов/месяц — это эквивалент одного полного рабочего дня, возвращённого к содержательной работе.
- **Средний** для широкой ML-аудитории (инженеры, которые хотели бы следить, но обычно не находят на это времени). Для них продукт переводит трекинг из "вообще не делаю" в "проверяю 5 мин утром" — эффект не в экономии времени, а в увеличении охвата.
- **Низкий** для тех, кто arXiv читает эпизодически и у кого нет одновременных долгоиграющих проектов — им достаточно раз в месяц открыть ручной listing.
